# DeskPilot 用户价值与差异化策略

## 1. 项目现状评估

### 1.1 已实现的能力

**PC Agent（完成度 ~75%）**

- 4 个执行器全部可用：Shell、Claude Code CLI、VS Code、浏览器
- 安全模型完整：44 条命令白名单、危险操作确认流程、HMAC 签名、审计日志
- TRTC 房间加入与屏幕共享（浏览器方式，1080p/720p 自适应）
- 意图分类：Claude CLI 优先，关键词匹配兜底
- WebSocket 重连（指数退避 1s → 30s）

**Cloud Orchestrator（完成度 ~70%）**

- 会话管理、TOTP 配对码、UserSig 生成
- TRTC Conversational AI Bot 生命周期管理
- Bot 回调路由 + LLM 意图分类（Groq 可选，关键词兜底）
- WebSocket 中继（Agent ↔ Cloud 双向）

**Mobile Client（完成度 ~20%）**

- React Navigation 路由、深色主题 UI
- TRTC 房间加入骨架
- 语音采集、屏幕显示、对话 UI 均为占位实现

**Shared Types（完成度 ~95%）**

- 11 种意图类型、4 种执行器、命令/结果载荷、WebSocket 消息格式定义完整

### 1.2 关键缺口

| 缺口 | 严重程度 | 影响 |
|------|---------|------|
| 手机端语音采集未实现 | **致命** | 核心语音管线断裂，端到端不可用 |
| 命令结果无法回传到手机 UI | 高 | 用户无法看到执行反馈 |
| 零自动化测试 | 高 | 质量无保障，重构风险大 |
| LLM 分类退化为关键词匹配 | 中 | 无 Groq API Key 时意图识别准确率低 |
| 无部署/CI 配置 | 中 | 无法快速上线验证 |
| 文档均为 TODO 占位 | 低 | 影响协作，不影响功能 |

### 1.3 端到端管线状态

```
[手机语音采集] ❌ 未实现
      ↓
[TRTC 房间传输] ✅ 可用
      ↓
[Conversational AI Bot ASR] ✅ 依赖 TRTC 服务
      ↓
[LLM 意图分类] ⚠️ Groq 可选，兜底为关键词
      ↓
[WebSocket → PC Agent] ✅ 可用
      ↓
[命令执行] ✅ 4 个执行器均可用
      ↓
[结果回传 → 手机显示] ❌ 未实现
```

**结论：后端管线基本通畅，两端（语音输入和结果展示）未接通。**

## 2. 竞品分析

### 2.1 直接竞品

#### Claude Code 移动端（Anthropic 官方）

- 文本指令控制电脑上的 Claude Code 会话
- 优势：官方产品、零部署成本、稳定可靠
- 劣势：纯文本交互，看不到电脑屏幕，无语音

#### WhatsApp / Telegram + OpenClaw 类方案

- 通过消息平台连接 AI Agent 操控电脑
- 优势：用户无需安装新 App，借助现有聊天平台的巨大用户基数
- 劣势：纯文本，高延迟，无屏幕共享，无语音

#### VS Code Remote / JetBrains Gateway

- IDE 远程开发方案
- 优势：完整 IDE 体验，行业标准
- 劣势：需要大屏设备，不适合手机，无 AI 语音交互

### 2.2 能力矩阵

| 能力 | Claude Code 移动端 | WhatsApp+OpenClaw | VS Code Remote | **DeskPilot** |
|------|-------------------|-------------------|----------------|---------------|
| 文本指令控制 | ✅ | ✅ | ✅ | ✅ |
| 语音控制 | ❌ | ❌ | ❌ | ✅* |
| 实时屏幕共享 | ❌ | ❌ | ✅（需大屏） | ✅ |
| AI 屏幕理解 | ❌ | ❌ | ❌ | ❌（可扩展） |
| 部署门槛 | 极低 | 低 | 中 | 高 |
| 延迟 | 秒级 | 秒级 | 毫秒级 | 毫秒级 |
| 用户基数 | Anthropic 生态 | 10亿+ | 千万级 | 零 |
| 非开发者可用 | ❌ | 部分 | ❌ | 可扩展 |

*语音控制当前为未完成状态

### 2.3 核心问题

**如果 DeskPilot 仅定位为"语音版 Claude Code 远程控制"，没有差异化。** Claude Code 官方移动端已经覆盖了核心场景（手机上给电脑发指令），且部署成本为零。DeskPilot 在这个赛道上没有胜算。

## 3. 差异化方向

DeskPilot 唯一有壁垒的技术组合是 **实时音视频 + 语音 + 屏幕共享**。以下三个方向利用了这个组合，且竞品无法轻易复制。

### 3.1 方向 A：屏幕感知 AI（推荐）

**核心主张：AI 看着你的屏幕和你对话。**

与其让用户描述问题（"第 42 行有个 bug"），不如让 AI 直接看到屏幕内容，结合语音理解用户意图。

**典型场景：**

- 用户看到报错，说"这个怎么修" → AI 看到屏幕上的错误信息，直接修复
- 用户说"把这个函数重构一下" → AI 看到光标所在的函数，执行重构
- 用户说"这个页面样式不对" → AI 看到浏览器渲染结果，修改 CSS

**技术路径：**

```
屏幕共享流 → 定时截图（每 2-5 秒）→ 视觉模型分析 → 上下文注入 LLM
                                         ↓
                              结合语音指令 → 生成精准命令
```

**所需新增能力：**

1. 屏幕截图采集（从 TRTC 视频流中抽帧）
2. 视觉模型接入（Claude Vision / GPT-4V）
3. 视觉上下文与语音意图融合的 Prompt 工程

**可行性：中等。** 视觉模型 API 已成熟，关键挑战是截图频率与 API 成本的平衡。

**壁垒：高。** 竞品（纯文本方案）无法获取屏幕内容，无法复制此能力。

### 3.2 方向 B：非开发者远程协助

**核心主张：帮不懂技术的人操作电脑。**

目标用户不是开发者，而是需要远程协助的普通人（父母、客户、同事）。

**典型场景：**

- 远程教父母："帮我打开那个 Excel，找到第三列"
- IT 支持："点左上角的设置按钮，然后选网络"
- 远程培训："打开浏览器，输入这个网址"

**技术路径：**

```
求助者手机 ←→ TRTC 房间 ←→ 被协助者电脑
                 ↑
        AI 助手（理解语音 + 看屏幕 → 引导操作或自动执行）
```

**所需调整：**

1. 执行器从"开发工具"扩展到"桌面操作"（鼠标点击、键盘输入、应用切换）
2. 意图类型从"编程任务"扩展到"桌面操作"（打开应用、点击位置、输入文字）
3. 简化配对流程（生成链接而非安装 Agent）

**可行性：高。** 技术栈已有（TRTC 屏幕共享 + 语音），主要是产品层面的调整。

**壁垒：中。** 远程桌面工具很多（TeamViewer、向日葵），但"AI 语音驱动"是新维度。

### 3.3 方向 C：实时协作编程教学

**核心主张：边看边说边操作的编程教学。**

导师或 AI 在一端讲解，学生在另一端看到实时屏幕操作，可以语音提问。

**典型场景：**

- AI 导师模式："我来教你写一个 React 组件，先看我怎么做"
- 结对编程："你来写，我看着，有问题我会提醒你"
- Code Review："我帮你看看这段代码" → AI 边看屏幕边语音讲解

**所需新增能力：**

1. AI 主动观察模式（持续分析屏幕内容，发现问题主动提醒）
2. 教学剧本引擎（按步骤引导，检查学生操作是否正确）
3. 语音对话记录与回放

**可行性：中。** 教学产品需要大量内容和 UX 打磨。

**壁垒：中。** 体验远优于文字教程，但需要持续投入内容。

## 4. 推荐策略

### 短期（1-2 周）：补齐核心管线 + 验证方向 A

1. **接通语音管线** — 实现移动端语音采集，打通端到端
2. **接通结果回传** — 命令执行结果显示在手机 UI 上
3. **原型验证屏幕感知** — 从 TRTC 流抽帧 → Claude Vision 分析 → 验证"看屏幕回答问题"的可行性

### 中期（3-4 周）：屏幕感知 MVP

1. **实现屏幕上下文融合** — 语音指令 + 屏幕截图 → 精准命令生成
2. **优化截图策略** — 事件驱动（屏幕变化时）而非定时，降低 API 成本
3. **添加核心测试** — 意图分类、命令执行、安全模块的单元测试
4. **部署管线** — Docker + CI/CD，可让外部用户试用

### 长期（2-3 月）：产品化

1. **扩展到非开发者场景** — 桌面操作执行器（鼠标、键盘、应用控制）
2. **多模态交互** — 手机触控指向屏幕位置 + 语音（"把这个移到那里"）
3. **Electron 原生屏幕采集** — 替代浏览器方案，提升性能和兼容性
